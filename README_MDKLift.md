# MDKLift - мобильная диспетчерская система для контроля лифтового оборудования

**MDKLift** - клиентское мобильное приложение для удалённого мониторинга состояния лифтов в составе комплекса диспетчерского контроля. Приложение расширяет возможности стационарной диспетчерской системы, предоставляя мобильный доступ к актуальным данным о состоянии лифтовых установок.

## О проекте

В большинстве классических систем диспетчеризации доступ к информации о состоянии лифтового оборудования сосредоточен на стационарном рабочем месте диспетчера. Это ограничивает мобильность персонала и затрудняет оперативное получение данных инженерами и мастерами, находящимися вне диспетчерского пункта.

**MDKLift** решает эту задачу за счёт мобильного клиента, который позволяет:

- получать сведения о состоянии закреплённых за пользователем лифтовых установок;
- просматривать активные телеметрические события (аварии, неисправности, отключения и информационные события);
- работать с последними синхронизированными данными при отсутствии соединения;
- использовать отдельные сценарии для диспетчера и инженера;
- переходить к сервисному приложению для технического обслуживания станции управления лифтом.

## Роли пользователей

### Диспетчер

Для диспетчера реализован режим непрерывного мониторинга состояния лифтовых установок. Данные автоматически обновляются при поступлении новых событий от серверной части.

Диспетчер может:

- просматривать список закреплённых лифтов;
- отслеживать активные события;
- фильтровать события по категориям;
- открывать карточку лифта с основной и дополнительной информацией;
- просматривать историю событий;
- выполнять поиск объекта по адресу;
- пользоваться справочником типов событий.

### Инженер / мастер

Для инженера реализован отдельный режим работы, ориентированный на выездное обслуживание объектов.

Инженер может:

- просматривать список закреплённых лифтовых установок;
- обновлять данные по запросу;
- открывать карточку объекта;
- строить маршрут до объекта через внешнее картографическое приложение;
- переходить в сервисный режим технического обслуживания станции управления лифтом.

## Основные возможности

### Мониторинг состояния лифтов

Главный экран отображает список лифтовых установок в виде адаптивной сетки карточек. Каждая карточка содержит:

- идентификатор и наименование лифта;
- модель и тип оборудования;
- адрес объекта;
- активные события;
- наиболее приоритетное событие;
- визуальную индикацию состояния.

### Фильтрация событий

События группируются по категориям и отображаются с цветовой индикацией:

| Категория | Назначение |
|---|---|
| Авария | Критические аварийные состояния |
| Неисправность | Ошибки и предупреждения оборудования |
| Отключение | События, связанные с потерей связи или отключением |
| Информация | Штатные и информационные события |

### Карточка лифтовой установки

Карточка лифта содержит подробную информацию об объекте:

- текущие активные события;
- приоритетное событие;
- технические характеристики;
- заводской номер;
- тип и модель лифта;
- грузоподъёмность;
- адрес установки;
- даты технических проверок и обслуживания.

Для пользователя с ролью инженера в карточке доступна кнопка построения маршрута до объекта.

### История событий

Приложение сохраняет историю появления и снятия событий. История может просматриваться:

- по конкретной лифтовой установке;
- по типу события;
- с отображением даты, времени, приоритета и действия.

### Поиск по адресу

Реализован поиск лифтовых установок по адресу. Поисковый запрос нормализуется, поэтому поиск менее чувствителен к регистру, лишним пробелам и частичному вводу.

### Справочник событий

В приложении предусмотрен справочник типов событий. Он помогает пользователю быстро определить значение и приоритет события, поступившего от лифтового оборудования.

### Работа при потере соединения

При отсутствии связи с сервером приложение не блокирует пользовательский интерфейс. На экране отображается уведомление о недоступности сервера, а данные продолжают показываться из локального хранилища.

## Эмуляция внешних функций

Часть внешних функций в проекте была **эмулирована**, так как разработка выполнялась на основе предоставленной документации и без постоянного доступа к промышленной серверной инфраструктуре.

В частности, были эмулированы:

- работа серверной части комплекса диспетчерского контроля;
- получение телеметрических данных от сервера;
- отдельные ответы API;
- сценарии отсутствия соединения с сервером;
- часть негативных сценариев, связанных с недоступностью внешних компонентов.

## Архитектура приложения

Приложение построено на основе многослойной архитектуры с разделением ответственности между слоями.

```text
Presentation  →  Domain  →  Data  →  Local DB / Remote API
```

### Presentation layer

Слой пользовательского интерфейса. Отвечает за отображение экранов, обработку действий пользователя и реактивное обновление данных.

Содержит:

- экраны авторизации;
- режим мониторинга;
- режим инженера;
- карточку лифта;
- историю событий;
- поиск по адресу;
- справочник событий;
- элементы навигации и виджеты интерфейса.

### Domain layer

Слой бизнес-логики. Содержит основные сущности и контракты репозиториев.

Содержит:

- сущности лифтовой установки;
- сущности событий;
- модели активных и исторических событий;
- интерфейсы репозиториев;
- бизнес-правила обработки данных.

### Data layer

Слой данных. Отвечает за взаимодействие с локальной базой данных и серверным API.

Содержит:

- DAO-классы;
- реализации репозиториев;
- API-клиент;
- сервис синхронизации;
- мапперы DTO в доменные сущности;
- таблицы Drift / SQLite.

## Локальное хранилище

В приложении используется локальная база данных SQLite. Работа с ней организована через библиотеку Drift.

Локальная база данных выполняет роль **Single Source of Truth** - единого источника достоверных данных для пользовательского интерфейса. Интерфейс получает данные не напрямую от сервера, а из локального хранилища. Серверная часть обновляет локальную базу при синхронизации, а UI автоматически реагирует на изменения.

Преимущества такого подхода:

- приложение остаётся работоспособным при временной потере связи;
- данные отображаются стабильно и согласованно;
- снижается зависимость интерфейса от сетевых запросов;
- упрощается обновление экранов при изменении данных;
- история событий сохраняется локально.

### Основные сущности локальной базы

- лифтовая установка;
- объект установки;
- техническая информация;
- активные события;
- история событий;
- типы событий;
- категории событий;
- пользовательские данные и параметры сессии.

## Интеграция с сервером

Взаимодействие с серверной частью организовано через REST API. Для выполнения HTTP-запросов используется библиотека Dio.

Общий цикл получения данных:

1. Пользователь вводит IP-адрес, порт, логин и пароль.
2. Приложение проверяет доступность сервера.
3. Выполняется авторизация.
4. Загружается корневая группа объектов.
5. Выполняется обход вложенных групп.
6. Загружается список доступных лифтов.
7. Для каждого лифта запрашиваются активные события.
8. Полученные данные сохраняются в локальную базу.
9. Интерфейс обновляется на основе данных из SQLite.

### Используемые API-запросы

| Метод | Endpoint | Назначение |
|---|---|---|
| `GET` | `/api/okean4-server/v1/ping` | Проверка доступности сервера |
| `GET` | `/api/okean4-server/v1/rootGroup` | Получение корневой группы объектов |
| `GET` | `/api/okean4-server/v1/children?parent=INT` | Получение дочерних групп |
| `GET` | `/api/okean4-server/v1/blocks?parent=INT` | Получение списка лифтов |
| `GET` | `/api/okean4-server/v1/events?block=INT` | Получение активных событий лифта |
| `GET` | `/api/okean4-server/v1/events?block=INT&start=TIME&end=TIME` | Получение событий за период |

По умолчанию для мобильного подключения используется порт `4005`.

## Интеграция с сервисным режимом

Для пользователей с ролью инженера предусмотрен переход в отдельное приложение сервисного режима - **СервисТул**.

Интеграция выполнена через механизм **deep linking**. Мобильная диспетчерская вызывает внешнее приложение по URI-схеме:

```text
servicetoolapp://open
```

Если приложение СервисТул не установлено, пользователю отображается диалоговое окно с предложением перейти на страницу загрузки.

Сервисный режим предназначен для технического обслуживания станции управления лифтом и включает:

- подключение к оборудованию по Wi-Fi, USB;
- offline-режим;
- просмотр диагностических параметров;
- изменение настроек станции управления;
- работу с EEPROM;
- обновление прошивки контроллера;
- интерпретацию кодов ошибок.

---

## Интеграция с картографическими сервисами

В режиме инженера реализовано открытие адреса лифтовой установки во внешнем картографическом приложении.

Логика открытия:

1. Сначала выполняется попытка открыть адрес в Яндекс.Картах.
2. Если Яндекс.Карты недоступны, используется системная geo-ссылка.
3. Если ни одно приложение не найдено, открывается web-версия Google Maps.

## Запуск проекта

### 1. Клонировать репозиторий

```bash
git clone https://github.com/qsshae-vi/mdk_lift.git
cd mdk_lift
```

### 2. Установить зависимости

```bash
flutter pub get
```

### 3. Сгенерировать файлы Drift

Так как в проекте используются `.g.dart`-файлы Drift, необходимо выполнить генерацию:

```bash
dart run build_runner build --delete-conflicting-outputs
```

### 4. Запустить приложение

```bash
flutter run
```

## Настройка подключения

При запуске приложения пользователь вводит параметры подключения к серверу:

- IP-адрес диспетчерского компьютера;
- порт;
- логин;
- пароль.

После успешной проверки соединения приложение сохраняет пользовательскую сессию и запускает синхронизацию данных.

**Для текущей версии достаточно ввести логин - диспетчер/инженер и пароль: 1234, ip-адрес в формате четырёх разделённых точками цифр и любой порт.**
